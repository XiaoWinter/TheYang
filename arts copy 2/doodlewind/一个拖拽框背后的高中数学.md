<p data-pid="wH3Mvhsr">很多时候一个看似简单的 bug 背后，都可能有着完全在作者意料之外的成因。这时候一路排查、调试和给出 fix 的过程往往可以相当神奇。最近我就遇到了个这样的问题，在此分享一下 :)</p><h2><b>缘起</b></h2><p data-pid="HC7h7YE0">最近我在维护一个用于平面设计的编辑器项目。在编辑器的画布上，图片是支持拖拽、旋转和裁切的，像这样：</p><figure data-size="normal"><img src="https://pica.zhimg.com/v2-6fd919acd60deca25db99d2d0b3dafd3_720w.gif?source=d16d100b" data-caption="" data-size="normal" data-rawwidth="298" data-rawheight="216" data-thumbnail="https://picx.zhimg.com/v2-6fd919acd60deca25db99d2d0b3dafd3_720w.jpg?source=d16d100b" class="content_image" width="298"></figure><p data-pid="R6w7KRIh">为了保证图片裁切后始终可见，我们需要限制用户的拖拽范围。对于普通的图片，下面这种边界限制显然很容易实现：</p><figure data-size="normal"><img src="https://pic1.zhimg.com/v2-89451ea2372e98e2c0484242f6bdabda_720w.gif?source=d16d100b" data-caption="" data-size="normal" data-rawwidth="452" data-rawheight="240" data-thumbnail="https://picx.zhimg.com/v2-89451ea2372e98e2c0484242f6bdabda_720w.jpg?source=d16d100b" class="origin_image zh-lightbox-thumb" width="452" data-original="https://picx.zhimg.com/v2-89451ea2372e98e2c0484242f6bdabda_720w.jpg?source=d16d100b"></figure><p data-pid="dI69G_wh">但是一旦图片存在旋转角度，这时的行为就显得很诡异了：</p><figure data-size="normal"><img src="https://picx.zhimg.com/v2-787f17b972f7ed7c65b52a69b362904e_720w.gif?source=d16d100b" data-caption="" data-size="normal" data-rawwidth="412" data-rawheight="244" data-thumbnail="https://pic1.zhimg.com/v2-787f17b972f7ed7c65b52a69b362904e_720w.jpg?source=d16d100b" class="content_image" width="412"></figure><p data-pid="VZzyGNrO">这显然不是预期的行为，那么该如何修复这个 bug 呢？</p><h2><b>瓶颈 1：祖传代码</b></h2><p data-pid="2I-ZTKHV">排查这个问题时，首先需要面对的是已有的代码实现。之前的代码虽然有类似的拖拽限制逻辑，但它所实现的 UI 交互和新版的需求有所不同（例如在拖拽时，我所需要移动的是图片，而旧版实现中移动的是裁剪框），故而没有办法直接复用。<b>但如果现成的代码拿来改改就能解决问题，那何苦自己重新搞一套呢？</b>本着这个再普通不过的想法，我首先尝试的是<b>搞懂现有代码的实现</b>。</p><p data-pid="o1DLNRlN">不读不知道，一读吓一跳。为了这个拖拽限制，现有的代码库中刨除掉各种胶水代码，还有 <b>150+</b> 行代码直接与这个限制的计算相关。为什么会这么复杂呢？这个实现的步骤大致是这样：</p><ul><li data-pid="01ZKp92b">对 0~90 度旋转，依次判断图片<b>左、上、右、下</b>四条边是否与裁切框的左、上、右、下相交。</li><li data-pid="PR4cVV5W">对 90~180 度旋转，依次判断图片<b>上、右、下、左</b>四条边是否与裁切框的左、上、右、下相交。</li><li data-pid="-0sugdZx">对 180~270 度旋转，依次判断图片<b>右、下、左、上</b>四条边是否与裁切框的左、上、右、下相交。</li><li data-pid="kA0OqBNy">对 270~360 度旋转，依次判断图片<b>下、左、上、右</b>四条边是否与裁切框的左、上、右、下相交。</li></ul><p data-pid="ovRQE124">这个实现确实可以说很符合直觉。然而同时，这个算法就有了 <code>4 x 4 = 16</code> 个可能的分支出口，每个出口里都有一系列相似但有区别的三角函数计算。虽然经过抽象封装，最后代码中只有四个用于实际计算的函数，但这个复杂度已经使得我很难通过小修小补的方式将它适配到新的交互方式上了。因此我决定花一些时间，思考如何<b>重写</b>。</p><h2><b>瓶颈 2：高中数学</b></h2><p data-pid="x8ypHWWw">既然决定了重写，那么核心的算法显然就可以另起炉灶重来了。和上面非常直接的这种直觉比起来，我在观察了这套交互之后，找到了另一种偷懒的直觉：只要你<b>把屏幕倾斜一下</b>，那么<b>旋转后</b>的情况就可以化归为<b>没有旋转时</b>的情况了呀！这也就是说，在代码实现上，<b>旋转后是有可能直接复用不存在旋转时的简单逻辑的</b>。听起来是不是省心了很多呢？</p><p data-pid="DJQxugmf">光有 idea 是不行的，把它实现出来才有意义。对于<b>把屏幕倾斜一下</b>这个 idea，它能够如何落实到代码实现上呢？高中数学的坐标系概念给了我灵感：一个点的位置，在多个不同的坐标系中可以有不同的表示。这样一来对于旋转后的图片，<b>只要我们将坐标系随之旋转</b>，那么在旋转后的坐标系中，计算拖拽限制应当就不是一件难事了。这套新思路可以总结为这样的算法：</p><ol><li data-pid="_i-9WjBq">当图片矩形存在旋转角 θ 时，我们将拖拽事件的 dx 和 dy 偏移量映射到和原始坐标系夹角 θ 的新直角坐标系上。</li><li data-pid="sZ2tO2mf">使用新坐标系上的偏移量 dx' 和 dy'，复用现有代码计算限制。</li><li data-pid="7qGroMJl">将添加了限制的 dx' 和 dy' 变换回 dx 和 dy，使用这两个校正后的偏移量来移动元素即可。</li></ol><p data-pid="NzcmhJoX">听起来「映射」和「变换」也不是件容易的事，而且我也<b>不确定这个算法是否是正确的</b>。如果吭哧吭哧实现完发现不能用，那么时间显然就浪费了。所以该怎么验证这个想法呢？我想到了个简单的方式：<b>取特殊值</b>。</p><p data-pid="34Ei6coi">旋转角为任意角度的时候，变换的公式需要推导。但是如果刚好旋转了 90 度或 180 度，这时的变换就十分简单，像 90 度时就是这样：</p><div class="highlight"><pre><code class="language-text"><span></span>// 正变换x' = y
y' = -x
// 逆变换x = -y'
y = -x'
</code></pre></div><p data-pid="WC8FhAqw">这显然非常容易通过小修小改现有代码的方式来实现。而实现后，对于旋转 90 度的图片，拖拽限制就这样神奇地改变了。这个尝试给了我很大的信心，因此我开始尝试推导一般情形下的变换，先根据直觉写出这个公式：</p><div class="highlight"><pre><code class="language-text"><span></span>x' = xcosθ + ysinθ
y' = xsinθ + ycosθ
</code></pre></div><p data-pid="6FvQkLZ1">然后我试图据此求出</p><div class="highlight"><pre><code class="language-text"><span></span>x = ?x' + ?y'
y = ?x' + ?y'
</code></pre></div><p data-pid="5If-8Nff">这个方程比较难直接通过高中数学暴力算出来，我尝试通过矩阵的变换来计算它，也就是求下面这个变换矩阵的逆矩阵：</p><div class="highlight"><pre><code class="language-text"><span></span>| cosθ sinθ | 
| sinθ cosθ |
</code></pre></div><p data-pid="T50-X9e-">但是在套用现成的矩阵变换公式的时候，这个矩阵的行列式可能为零，而这时候逆矩阵不存在……我对此感到匪夷所思，于是厚着脸皮请教了正在 T 大数学系读博的敏神，敏锐的敏神一眼就指出了问题：变换矩阵里有个值应该是负的……果然毕业以后太多东西都还回去了啊[捂脸]</p><p data-pid="lyfMYVAG">改了改以后答案变成了这样（求忽略不支持 LaTeX 的丑陋写法）：</p><div class="highlight"><pre><code class="language-text"><span></span>| x' | = |  cosθ sinθ | | x  |
| y' |   | -sinθ cosθ | | y  |

| x  | = | cosθ -sinθ | | x' |
| y  |   | sinθ  cosθ | | y' |
</code></pre></div><p data-pid="9an7ljR4">把这个正逆变换的公式应用到现在的代码上，就得到了这样的效果：</p><figure data-size="normal"><img src="https://pic1.zhimg.com/v2-2fb9a85323cd236bb4fca9e8b80fe0ea_720w.gif?source=d16d100b" data-caption="" data-size="normal" data-rawwidth="412" data-rawheight="244" data-thumbnail="https://picx.zhimg.com/v2-2fb9a85323cd236bb4fca9e8b80fe0ea_720w.jpg?source=d16d100b" class="content_image" width="412"></figure><p data-pid="OvHVXb3y">看起来好像大功告成了啊！可惜这还不是终点……</p><h2><b>瓶颈 3：一步之遥</b></h2><p data-pid="1SghvmVR">本来问题似乎已经解决了，但是在合并代码前的自测时，却发现旋转后的拖拽限制可能会出现一个莫名其妙的固定偏移量：</p><figure data-size="normal"><img src="https://picx.zhimg.com/v2-b59508ff85f02ac82b693e8d16c51e75_720w.gif?source=d16d100b" data-caption="" data-size="normal" data-rawwidth="412" data-rawheight="244" data-thumbnail="https://picx.zhimg.com/v2-b59508ff85f02ac82b693e8d16c51e75_720w.jpg?source=d16d100b" class="content_image" width="412"></figure><p data-pid="DVKxtkDq">这就让人头大了……算法看起来是正确的，一般情况和若干特殊情况下的效果也是正确的，但是少数情形下却有这么大的误差，实在是非常诡异。我依次排查了<b>新加入的代码</b>和用于获得偏移量的<b>胶水代码</b>，都没有找到问题所在。因为这个 bug，我不得不暂时放下了这个重构，优先处理一些其它的细节需求。</p><p data-pid="ugmPj-pJ">有意思的是，即便放下了一个问题，对它的思考说不定也在默默地继续。某次浴室沉思的时候，我想到了一个被忽略的地方：<b>我一直不愿意改动的「简单逻辑限制」代码</b>。</p><p data-pid="gULhDw8D">我们一开始就提到过没有旋转时的拖拽限制非常好写，就像 GLSL 里面「掐头去尾」的 <code>clamp</code> 函数：</p><div class="highlight"><pre><code class="language-text"><span></span>const clamp = (x, lower, upper) =&gt; Math.max(lower, Math.min(x, upper))
dx = clamp(dx, minLeft, maxLeft)
dy = clamp(dy, minTop, maxTop)
</code></pre></div><p data-pid="2BvH8GzE">这个 <code>clamp</code> 本身是正确的，因此我也一直认为这段代码是正确无误的。但考虑了「旋转」这个因素后，<code>left</code> 和 <code>top</code> 的来源是否值得信任呢？它们是在屏幕坐标系下的偏移量，求出它们的代码非常简单，大概这样：</p><div class="highlight"><pre><code class="language-text"><span></span>minLeft = rect.left - rect.width
maxTop = rect.top + rect.height
// ...
</code></pre></div><p data-pid="xBbLdOJ5">一个元素在浏览器内的位置，是相对于屏幕左上角的。但上文中的变换公式中，位置是相对于拖拽框中心点的。考虑这一因素之后，这几个变量的有效性就存疑了。对此我的尝试是：基于两个矩形<b>中心点之间的距离</b>去计算拖拽限制，而非直接利用现成的偏移量。由于中心点的间距抹除了初始位置对计算的影响，那么偏移量就应当是可以消除的。重构之后的代码用 <code>centerDeltaX</code> 和 <code>centerDeltaY</code> 替代了上面的中间变量，得到的效果如下所示：</p><p><br></p><figure data-size="normal"><img src="https://picx.zhimg.com/v2-b4b7058fa64a4efcbfa1565f91828ddb_720w.gif?source=d16d100b" data-caption="" data-size="normal" data-rawwidth="402" data-rawheight="236" data-thumbnail="https://pica.zhimg.com/v2-b4b7058fa64a4efcbfa1565f91828ddb_720w.jpg?source=d16d100b" class="content_image" width="402"></figure><p><br></p><p data-pid="-kBpxRmy">于是，最麻烦的 bug 就这样修复了~这个改进的收益还是有的：150+ 行的代码被优化到了 10+ 行的量级，代码执行路径上的分支也从 16 个优化到了 0 个。最后的版本如下所示：</p><div class="highlight"><pre><code class="language-js"><span></span><span class="c1">// 在高阶计算函数中缓存 sin 与 cos</span>
<span class="kr">const</span> <span class="nx">rotateVector</span> <span class="o">=</span> <span class="nx">utils</span><span class="p">.</span><span class="nx">getVectorRotator</span><span class="p">(</span><span class="nx">element</span><span class="p">.</span><span class="nx">rotate</span><span class="p">);</span>
<span class="kr">const</span> <span class="p">{</span> <span class="nx">minLeft</span><span class="p">,</span> <span class="nx">maxLeft</span><span class="p">,</span> <span class="nx">minTop</span><span class="p">,</span> <span class="nx">maxTop</span><span class="p">,</span> <span class="nx">centerDeltaX</span><span class="p">,</span> <span class="nx">centerDeltaY</span> <span class="p">}</span> <span class="o">=</span> <span class="nx">element</span><span class="p">.</span><span class="nx">$getDragLimit</span><span class="p">();</span>
<span class="c1">// 变换至旋转后坐标系</span>
<span class="c1">// 带 _ 后缀的变量处于旋转后参考系中</span>
<span class="kr">const</span> <span class="p">[</span><span class="nx">dx_</span><span class="p">,</span> <span class="nx">dy_</span><span class="p">]</span> <span class="o">=</span> <span class="nx">rotateVector</span><span class="p">(</span><span class="nx">dx</span><span class="p">,</span> <span class="nx">dy</span><span class="p">);</span>
<span class="c1">// 最终偏移量 deltaX = 拖拽事件偏移量 dx + 两矩形中心点距离</span>
<span class="nx">centerDeltaXconst</span> <span class="p">[</span><span class="nx">centerDeltaX_</span><span class="p">,</span> <span class="nx">centerDeltaY_</span><span class="p">]</span> <span class="o">=</span> <span class="nx">rotateVector</span><span class="p">(</span><span class="nx">centerDeltaX</span><span class="p">,</span> <span class="nx">centerDeltaY</span><span class="p">);</span>
<span class="kr">const</span> <span class="nx">clampedDeltaX_</span> <span class="o">=</span> <span class="nx">utils</span><span class="p">.</span><span class="nx">clamp</span><span class="p">(</span><span class="nx">centerDeltaX_</span> <span class="o">+</span> <span class="nx">dx_</span><span class="p">,</span> <span class="nx">minLeft</span><span class="p">,</span> <span class="nx">maxLeft</span><span class="p">);</span>
<span class="kr">const</span> <span class="nx">clampedDeltaY_</span> <span class="o">=</span> <span class="nx">utils</span><span class="p">.</span><span class="nx">clamp</span><span class="p">(</span><span class="nx">centerDeltaY_</span> <span class="o">+</span> <span class="nx">dy_</span><span class="p">,</span> <span class="nx">minTop</span><span class="p">,</span> <span class="nx">maxTop</span><span class="p">);</span>
<span class="c1">// 将修正后偏移量反变换回原始坐标系</span>
<span class="kr">const</span> <span class="p">[</span><span class="nx">clampedDeltaX</span><span class="p">,</span> <span class="nx">clampedDeltaY</span><span class="p">]</span> <span class="o">=</span> <span class="nx">rotateVector</span><span class="p">(</span><span class="nx">clampedDeltaX_</span><span class="p">,</span> <span class="nx">clampedDeltaY_</span><span class="p">,</span> <span class="kc">true</span><span class="p">);</span>
<span class="p">[</span><span class="nx">dx</span><span class="p">,</span> <span class="nx">dy</span><span class="p">]</span> <span class="o">=</span> <span class="p">[</span><span class="nx">clampedDeltaX</span> <span class="o">-</span> <span class="nx">centerDeltaX</span><span class="p">,</span> <span class="nx">clampedDeltaY</span> <span class="o">-</span> <span class="nx">centerDeltaY</span><span class="p">];</span>
<span class="p">[</span><span class="nx">left</span><span class="p">,</span> <span class="nx">top</span><span class="p">]</span> <span class="o">=</span> <span class="p">[</span><span class="nx">drag</span><span class="p">.</span><span class="nx">left</span> <span class="o">+</span> <span class="nx">dx</span><span class="p">,</span> <span class="nx">drag</span><span class="p">.</span><span class="nx">top</span> <span class="o">+</span> <span class="nx">dy</span><span class="p">];</span>
</code></pre></div><h2><b>总结</b></h2><p data-pid="4DHNl3Jy">到此为止，一段折腾的故事终于告一段落了。虽然这个需求未必是我们日常开发中可能遇到的，但调试过程中的一些总结感觉还是有些参考价值的：</p><ul><li data-pid="C77a_2pX">直觉还是很重要。譬如敏神的直觉就可以直接指出我在关键的地方少了个负号（致谢致谢），而写工程代码的直觉，可能也就是尽可能地依赖现有的工具找捷径解决问题吧 XD</li><li data-pid="RjsBhiaa">对复杂的问题，把代码逻辑梳理正确比起瞎改变量然后保存反复尝试，要靠谱得多。</li><li data-pid="iZ5UVEt6">重新实现一套逻辑显得很麻烦的时候，可以使用特殊的输入输出来给出 POC 的原型实现，这还有助于放大问题与提供干净的复现环境。</li><li data-pid="SVomFxqn">注意你觉得毫不起眼的角落，整个执行链路上的代码都值得纳入考虑。</li><li data-pid="aKKyZQln">很多技术问题一路钻到底就能得到答案。我也可以选择搁置这个优化，但这样就错过了一个锻炼的机会 :)</li></ul><p data-pid="hdvk1x0x">限于我的水平，这段调试经历显得有些曲折。希望对感兴趣的同学有所帮助~</p>
